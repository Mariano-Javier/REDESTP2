Sistema de Gestión de Tareas

Arquitectura Cliente-Servidor con Flask y SQLite

Python 3.10+ Flask SQLite3

Estructura del Proyecto

Organización de los archivos para garantizar modularidad y separación de responsabilidades: 

REDESTP2/
├── venv/                  # Entorno virtual (Aislamiento de recursos)
├── img/                   # Imagenes del proyecto
├── templates/             # Plantillas HTML (Capa de presentación)
│   └── bienvenida.html    # Documentación y guía de uso
├── cliente.py             # Cliente de Consola (Interactúa con la API)
├── servidor.py            # Servidor Flask (Backend y lógica de negocio)
├── database.py            # Script para inicializar tablas en SQLite
├── gestion_tareas.db      # Base de datos persistente
└── requirements.txt       # Librerías necesarias (Flask, Requests)

Interoperabilidad del Sistema

Una de las ventajas de esta arquitectura es que el Servidor Flask expone una interfaz estandarizada (API REST). Esto permite obtener el mismo resultado utilizando diferentes clientes:

Cliente de Consola (Python)

Usando un menú dentro de la terminal. Utiliza la librería requests para comunicarse de forma automática.

Uso Simplificado

Herramientas de API (Thunder Client)

Permite testear cada endpoint manualmente enviando paquetes JSON.

Uso Técnico / Testeo

Conclusión técnica: Independientemente del cliente utilizado, la lógica de negocio y la persistencia en SQLite ocurren siempre en el servidor. Esto garantiza que los datos sean consistentes: si se registra un usuario en Thunder Client, se puede hacer el login de este con el menu de la consola.

Guía de Ejecución

1 Preparación del Entorno Virtual

Utilizamos un entorno de ejecución aislado e instalamos las dependencias: 

python -m venv venv
.\venv\Scripts\activate  # En Windows
pip install -r requirements.txt

2 Inicialización de la Base de Datos

Creamos las tablas de usuarios y tareas en SQLite antes de iniciar el sistema: 

python database.py

3 Lanzamiento del Sistema

Primero inicializamos el servidor y luego el cliente:

Terminal 1: Iniciar el Backend.

python servidor.py

Terminal 2: Iniciar la interfaz de usuario. 

python cliente.py

Manual del Cliente (cliente.py)

El cliente de consola permite gestionar los recursos del servidor mediante un menú interactivo:

1. Registrarse

Solicita usuario y clave. El servidor genera un hash seguro para la base de datos.

2. Iniciar Sesión

Valida credenciales. Si es exitoso, el servidor retorna el ID de usuario necesario para ver las tareas.

Visualización Automática

Al loguearse con éxito, el cliente invoca automáticamente a la función mostrar_tareas(user_id), que realiza una petición GET para listar las tareas asociadas a ese ID específico.

3. Salir

Finaliza el proceso del cliente y libera el socket de conexión.

Flujo de Comunicación

El sistema opera bajo un modelo distribuido sincronizado:

  • Seguridad: Las contraseñas se almacenan mediante algoritmos de hasheo (Werkzeug).
  • Autenticación: El servidor valida el hash y devuelve un user_id para sesiones dinámicas.
  • Recursos: El cliente solicita tareas específicas mediante peticiones HTTP GET y POST.

Respuestas Conceptuales

¿Por qué hashear contraseñas?

Es una medida de seguridad crítica para proteger la infraestructura. Al aplicar un hash (mediante Werkzeug), la contraseña real nunca se almacena en texto plano.

  • Privacidad: Ni el administrador de la base de datos puede ver las claves.
  • Seguridad: Si el archivo .db se filtra, los atacantes solo obtienen huellas digitales criptográficas imposibles de revertir.

Ventajas de usar SQLite

SQLite es el motor de persistencia ideal para este proyecto por su eficiencia y simplicidad técnica.

  • Persistencia: Los datos se mantienen guardados permanentemente en el disco, a diferencia de usar variables en memoria.
  • Portabilidad: No requiere configuración de servidores externos; la base de datos es un archivo más dentro del repositorio.
  • Ligereza: Consume recursos mínimos, ideal para sistemas distribuidos educativos.
